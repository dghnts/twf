# 要件定義書

## 1. 機能概要

TWF（Terminal Wallpaper Fit）は、ターミナルの背景画像を解析し、その色調に基づいてターミナルのカラースキームとフォント設定を自動的に最適化するCLIツールです。

本ツールは、背景画像の主要な色を抽出し、視認性を最大化するための最適なフォアグラウンドカラー、アクセントカラー、フォントスタイルを自動的に選択します。クロスプラットフォーム対応で、OS（Linux、macOS、Windows）やシェル環境（bash、zsh、fish等）に依存せず動作します。

ユーザーは単一のコマンドを実行するだけで、ターミナル環境を背景画像に調和した美しく読みやすい設定に変更できます。

## 2. 機能要件

### 2.1 画像解析機能

- ターミナルの背景画像ファイルを読み込み、画像の主要な色を抽出できること
- 画像の明度（明るさ）を分析し、背景が明るいか暗いかを判定できること
- 画像の彩度と色相を分析し、支配的な色調を特定できること
- 対応画像形式：PNG、JPEG、GIF、BMP、WebP
- 画像解析は3秒以内に完了すること
- 背景画像が設定されていない場合、ターミナルの現在の背景色を検出できること
- ANSIエスケープシーケンス（OSC 11）を使用してターミナルから背景色を取得できること
- 検出した背景色に基づいて、視認性の高いカラースキームを生成できること
- 背景色の検出に失敗した場合は、デフォルトの背景色（黒または白）を仮定して処理を継続できること

### 2.2 カラースキーム最適化機能

- 背景画像の色調に基づいて、視認性の高いフォアグラウンドカラーを自動選択できること
- ANSI 16色（基本8色 + 明るい8色）の最適な配色を生成できること
- 256色パレットに対応したターミナルでは、より詳細な色調整を提供できること
- True Color（24bit色）対応ターミナルでは、最高品質の色調整を提供できること
- コントラスト比がWCAG 2.1 AA基準（4.5:1以上）を満たすこと

### 2.3 フォント設定最適化機能

- 背景画像の複雑さに応じて、適切なフォントウェイト（太さ）を推奨できること
- 背景画像の明度に応じて、フォントの太さを調整できること
- システムにインストールされている等幅フォントを検出できること
- 推奨フォント設定をユーザーに提示できること

### 2.4 クロスプラットフォーム対応

- Linux、macOS、Windowsの各OSで動作すること
- bash、zsh、fish、PowerShellの各シェル環境に対応すること
- 各シェルの設定ファイル（.bashrc、.zshrc、config.fish等）を自動検出できること
- 各シェルの設定ファイルに適切な形式で設定を書き込めること

### 2.5 設定適用機能

- 生成したカラースキームを現在のターミナルセッションに即座に適用できること
- 設定を永続化するために、シェル設定ファイルに書き込めること
- 既存の設定をバックアップしてから新しい設定を適用できること
- 設定を元に戻す（ロールバック）機能を提供できること

### 2.6 プレビュー機能

- 設定を適用する前に、プレビューを表示できること
- プレビューには、各ANSI色のサンプルテキストを含むこと
- プレビューには、実際のコマンド出力例（ls、git status等）を含むこと

### 2.7 CLI インターフェース

- シンプルで直感的なコマンドラインインターフェースを提供すること
- 画像ファイルパスをオプション引数として受け取れること（必須ではない）
- 画像パスが指定されていない場合は、自動検出→背景色解析の順で処理すること
- `--image <PATH>` オプションで明示的に画像パスを指定できること
- `--detect` オプションで自動検出を強制できること
- `--color <COLOR>` オプションで背景色を直接指定できること（例：`--color "#1e1e1e"`）
- オプションフラグで動作をカスタマイズできること（--preview、--apply、--rollback等）
- ヘルプメッセージとバージョン情報を表示できること
- ヘルプメッセージには各オプションのユースケース例を含むこと
- `--image` オプションのヘルプには、自動検出が失敗した場合や事前にテストしたい場合に使用することを明記すること
- `--color` オプションのヘルプには、背景画像を使用していない場合や特定の背景色に合わせたい場合に使用することを明記すること
- エラーメッセージは日本語と英語の両方に対応すること

### 2.8 背景画像自動検出機能

- ターミナルエミュレータの設定ファイルから背景画像のパスを自動検出できること
- 対応ターミナル：iTerm2、Alacritty、Windows Terminal、GNOME Terminal、Kitty、WezTerm
- iTerm2の場合、`~/Library/Preferences/com.googlecode.iterm2.plist` から背景画像パスを取得できること
- Alacrittyの場合、`~/.config/alacritty/alacritty.yml` または `alacritty.toml` から背景画像パスを取得できること
- Windows Terminalの場合、`%LOCALAPPDATA%\Packages\Microsoft.WindowsTerminal_*\LocalState\settings.json` から背景画像パスを取得できること
- GNOME Terminalの場合、dconf/gsettingsから背景画像設定を取得できること
- 現在実行中のターミナルエミュレータを自動判定できること（環境変数 `TERM_PROGRAM`、`TERMINAL_EMULATOR` 等を使用）
- 画像パスが引数で指定されていない場合、自動検出を試みること
- 自動検出に失敗した場合は、背景色解析にフォールバックすること
- 複数のターミナル設定が見つかった場合、現在のターミナルに最も関連性の高い設定を優先すること
- 検出した画像パスをユーザーに通知すること（詳細モードの場合）

## 3. 非機能要件

### 3.1 パフォーマンス

- 画像解析から設定生成まで、合計5秒以内に完了すること
- メモリ使用量は100MB以下に抑えること
- 大きな画像（4K解像度以上）でも適切に処理できること

### 3.2 可用性

- ネットワーク接続なしで完全に動作すること（オフライン動作）
- 外部依存関係を最小限に抑えること
- 単一のバイナリファイルとして配布できること

### 3.3 保守性

- コードは明確にモジュール化されていること
- 各モジュールは単一責任の原則に従うこと
- ユニットテストのカバレッジは80%以上であること
- ドキュメントは日本語と英語の両方で提供すること

### 3.4 ユーザビリティ

- 初回実行時に簡単なチュートリアルを表示すること
- エラーメッセージは具体的で、解決方法を示すこと
- 進行状況をプログレスバーで表示すること

### 3.5 互換性

- POSIX準拠のシステムで動作すること
- 最小限のシステム要件で動作すること（古いOSバージョンでも可能な限り対応）
- ターミナルエミュレータの種類に依存しないこと（iTerm2、Alacritty、Windows Terminal、GNOME Terminal等）

### 3.6 セキュリティ

- ユーザーの設定ファイルを変更する前に、必ず確認を求めること
- バックアップファイルは安全な場所に保存すること
- ファイルパスのトラバーサル攻撃を防ぐこと

## 4. 制約事項

### 4.1 技術的制約

- 実装言語：Rust（クロスプラットフォーム対応とパフォーマンスのため）
- 画像処理ライブラリ：image crateを使用
- カラー解析：palette crateまたは同等のライブラリを使用
- CLI フレームワーク：clap crateを使用

### 4.2 環境制約

- ターミナルがANSIエスケープシーケンスに対応していること
- 背景画像を設定できるターミナルエミュレータであること
- ファイルシステムへの書き込み権限があること

### 4.3 リソース制約

- 初期バージョンでは、静的な背景画像のみをサポート（動画や動的背景は対象外）
- 初期バージョンでは、単一の背景画像のみを解析（複数画像の同時処理は対象外）
- GUIは提供せず、CLIのみとする

### 4.4 開発制約

- 開発期間は考慮しない（品質を優先）
- オープンソースとして公開することを前提とする
- ライセンス：MIT または Apache 2.0
